# TQAI Pro v2.9.7 — 專業投資角度覆核與修改紀錄

本次是以「專業投資身分」對既有 v2.9.6 程式碼進行全面檢視後的修改。
覆核範圍涵蓋全部 26 個引擎（`engines/*.py`）與 `app.py`；多數引擎（尤其
risk_engine / strategy_engine / decision_engine / chip_engine /
sector_rotation_engine / macro_engine / portfolio_engine 等）在既有版本
已經做得相當完整、且延續一貫的「誠實揭露限制」風格，因此本次修改集中在
**實際發現的三個具體缺口**，而非重寫既有邏輯——重寫已經寫得好的東西沒有
專業投資價值，找出真正缺的一塊才有。

## 為什麼是這三塊（專業投資角度的判斷）

一個專業投資人在使用這類系統時，除了「這檔股票的技術面/基本面訊號好不好」，
還會反射性地問三個問題，而這三個問題在 v2.9.6 都還沒有被回答：

1. **「就算訊號很好，我進得去、出得來嗎？」→ 流動性風險**
   原本的風險中心（RiskEngine）只涵蓋波動率、回撤、Beta、VaR，完全沒有
   成交量／成交值角度的流動性檢查。台股中小型股/興櫃股常見「線型漂亮但
   量能稀薄」的陷阱，訊號再好，量不夠時實際下單會有嚴重滑價，甚至根本
   沒有對手盤。

2. **「這幾個引擎的判斷，最後有沒有真的匯總成一個意見？」→ 共識儀表板的
   資料缺口**
   ChipEngine（三大法人籌碼）與 SectorRotationEngine（產業輪動）都已經
   存在且運作正常，但 DecisionEngine 的共識儀表板完全沒有讀取這兩個引擎
   的結果——等於「算了但沒被用上」，跟本專案自己在 momentum_engine.py /
   strategy_engine.py 反覆點名並修正過的同一類問題（訊號算出來卻沒真正
   影響最終判斷）。

3. **「這波上漲，是真買盤、還是散戶融資堆出來的？」→ 融資餘額趨勢**
   ChipEngine 原本只有「今天融資變化多少」的單日快照，沒有「這段期間
   融資餘額累積變化多少」的趨勢——後者才是判斷「這波漲勢的籌碼結構穩不
   穩」的關鍵。這是台股實務上（尤其中小型飆股）非常經典的風險檢查角度。

## 具體修改清單

### 1. `engines/risk_engine.py` — 新增流動性風險模組
- 新增 `add_liquidity_metrics()`：計算近20日平均成交值（新台幣），
  分三級（🔴極低流動性 / 🟡流動性偏低 / 🟢流動性正常）。
- `build_risk_report()` 整合流動性到 `risk_flags` 與綜合 `risk_level`
  評分（極低流動性額外計1風險點）。
- 門檻採業界經驗法則（日均成交值 <1,000萬元 / <5,000萬元），已在程式碼
  註解中明確標注這是經驗法則、非監管定義。

### 2. `engines/chip_engine.py` — 新增融資餘額趨勢
- 新增 `get_margin_trend(stock_code, lookback_days=20)`：逐日查詢 TWSE
  融資融券資料，計算融資餘額變化率，超過 ±20% 給出雙向解讀的提示文字
  （槓桿升溫 vs 去化沉澱/恐慌斷頭，不做單一方向斷定）。
- ⚠️ 效能考量：這個查詢需要逐日呼叫 TWSE API（約20次請求），刻意做成
  `app.py` 裡的「按鈕觸發」而非自動執行，且明確建議不要放進批次掃描。

### 3. `engines/decision_engine.py` — 共識儀表板擴充至9個維度
- 新增「籌碼面」投票（讀取 ChipEngine 三大法人合計買賣超方向）。
- 新增「產業輪動」投票（讀取 SectorRotationEngine 該股所屬產業的輪動
  訊號；需要使用者已在產業輪動頁面計算過，否則誠實顯示「資料不足」）。
- 新增「流動性」否決維度（跟既有的「風險等級」否決邏輯並列，任一觸發
  即為 NO TRADE）。
- 新增「總經背景」附帶顯示（MacroEngine 的旗標）——刻意**不**計入偏多/
  偏空票數，因為總經是全市場共同的環境變數，不是個股專屬訊號，混進票數
  容易誤導。

### 4. `app.py` — 對應的 UI 串接
- 風險中心新增「💧 流動性」欄位顯示。
- 籌碼中心新增「融資餘額趨勢」按鈕（避免自動觸發拖慢頁面）。
- 共識儀表板呼叫改為傳入 `chip_report` / `sector_signal` / `macro_flags`
  / `liquidity_level`，並新增總經背景的摺疊顯示區塊。
- 版面標題版本號同步更新為 v2.9.7（原本停留在過時的 v2.5 字樣）。

## 刻意不做的事情（維持既有的誠實揭露原則）

- 沒有新增任何需要付費資料源、新聞情緒、選擇權 Greeks 等本專案既有文件
  已多次說明「資料源不可得」的功能——這些限制沒有改變，本次修改沒有
  繞過或假造資料。
- 融資餘額趨勢刻意不做成自動批次功能，避免對 TWSE 造成不合理的請求量。
- 「產業輪動」與「總經背景」在共識儀表板中都設計成「沒算過就誠實顯示
  資料不足」，不會為了畫面好看而假裝有訊號。

## 尚未觸及、但值得未來繼續深入的方向（留給下次討論）

- ~~PortfolioEngine 尚未串接「流動性」作為配置權重的額外限制~~ → 已於
  v2.9.8 補上，見下方。
- ~~BacktestEngine 尚未把流動性納入回測~~ → 已於 v2.9.8 補上參與率檢查
  （融資餘額趨勢因為需要逐日 TWSE 請求，成本過高，暫緩納入回測）。
- ChipEngine 的融資餘額趨勢目前僅支援單一個股查詢，尚未有「全市場融資
  餘額變化排行榜」，如果有需求可以延伸（逐日全市場快照本來就會抓到，
  差別只在於要為監控名單中每一檔股票額外保留一份序列，工程量不小，
  這次沒有做）。
- compare_strategies() 的比較表尚未把新增的「參與率過高筆數」欄位納入
  並排比較（目前只在單一策略的 run_backtest 摘要中顯示），如果要在
  「策略A vs 策略B」的比較情境下也看到流動性真實性差異，可以延伸。

# TQAI Pro v2.9.8 — 專業投資角度覆核（第二輪）

延續 v2.9.7 的檢視方向，這次針對上一輪自己列出的「尚未觸及」項目繼續
補齊，一樣不重寫已經寫得好的既有邏輯，只補實際缺口。

## 為什麼是這兩塊

v2.9.7 補上了「流動性風險」的量測（RiskEngine），但只做到「量出來、
顯示出來」，還沒有真正影響到後續的兩個實務決策：
1. **投組建構會不會真的選到這種股票？** ——會，因為 PortfolioEngine
   原本完全沒讀取「流動性」欄位。
2. **回測績效會不會被這種股票的不切實際成交撐起來？** ——會，因為
   BacktestEngine 原本沒有檢查任何一筆交易的部位相對真實成交量合不合理。

這兩塊都是「算出來的風險指標，沒有真正接回決策鏈」的同一類缺口（跟
v2.9.7 補「籌碼面／產業輪動沒接回共識儀表板」是同一個思路的延伸）。

## 具體修改清單

### 1. `engines/scanner_engine.py` — 掃描結果新增流動性欄位
- Pipeline 呼叫 `RiskEngine.add_liquidity_metrics()`（只用既有的
  close/volume，不增加外部請求）。
- `scan()` 回傳結果新增「流動性」「日均成交值(百萬)」兩欄，讓下游的
  PortfolioEngine 與使用者都看得到。

### 2. `engines/portfolio_engine.py` — 投組建構新增流動性排除
- `build_portfolio()` 新增 `exclude_illiquid: bool = True` 參數：預設會
  把候選名單中「🔴 極低流動性」的股票排除，不會因為 AI Score 高就選入
  組合；若 `result_df` 沒有「流動性」欄位（例如舊版掃描結果），會誠實
  跳過檢查並在 note 中註明，不假裝篩選過。
- `weights_table` 新增顯示「流動性」欄位；回傳值新增 `illiquid_excluded`
  （被排除的股票清單），供 UI 呈現。
- 若候選股在排除極低流動性標的後變成空集合，會明確回傳空結果並說明
  原因，而不是默默回傳一個空的投資組合。

### 3. `engines/backtest_engine.py` — 回測新增參與率真實性檢查
- 新增 `max_participation_pct: float = 10.0` 參數（經驗法則，非監管
  規定）。每筆買進成交時記錄「參與率」= 買進股數 ÷ 當天真實成交量。
- `summary` 新增 `liquidity_realism` 欄位：平均/最高參與率、參與率過高
  的筆數與佔比、以及一段白話說明——如果回測績效主要靠「部位遠超過當天
  真實成交量」的交易撐起來，這種績效在真實市場很難被完整執行，此檢查
  只做揭露，不會回頭調整撮合價格或報酬率。

### 4. `app.py` — 對應的 UI 串接
- 投組建構頁新增「排除極低流動性標的」核取方塊（預設勾選）與被排除
  股票清單的顯示區塊。
- 回測結果面板新增「平均參與率／最高單筆參與率／參與率過高筆數」三個
  指標，參與率過高筆數 > 0 時用 `st.warning` 強調（而非普通提示文字）。

## 刻意不做的事情

- 沒有把參與率檢查回頭影響回測的撮合價格或報酬率——那需要真正的市場
  衝擊模型（例如根據參與率調整滑價），複雜度與資料需求都高出一個量級，
  這次選擇「如實揭露，不假裝修正」的做法，跟本專案一貫的誠實揭露原則
  一致。
- 沒有把融資餘額趨勢（v2.9.7 新增）納入回測——需要逐日 TWSE 請求，
  對長期回測（例如兩年 = 約 500 個交易日）成本過高，不合理。
